## Why this matters

The entire build/test/lint/format toolchain — Vite, Vitest, and the `vite-plus` CLI (which bundles oxlint, oxfmt, the bundler, and the test runner) — is specified with floating `@latest`/`latest` version ranges against a pre-1.0 (`0.x`) package. CI is reproducible _today_ only because it runs `bun install --frozen-lockfile`. But any non-frozen install — a developer's local `bun install`, or Dependabot regenerating the lockfile — silently pulls the newest `0.x` with no semver-stability guarantee and no manifest signal of the intended version. For a `0.x` toolchain that owns the whole pipeline, this is a real reproducibility + supply-chain exposure. Pinning to the already-resolved version is a no-op for CI and stops silent drift.

## Steps

### Step 1: Determine the currently-resolved versions

Run `grep -n "@voidzero-dev/vite-plus-core\|@voidzero-dev/vite-plus-test\|\"vite-plus\"" bun.lock` and record the exact resolved version for each of the three packages. They are expected to share a version (e.g. `0.1.24`) but confirm each independently. Call the resolved versions `CORE_V`, `TEST_V`, `CLI_V`.

**Verify**: you have three concrete version strings, none containing `latest`.

### Step 2: Pin the specs in root `package.json`

Replace the floating specs with the exact resolved versions, in BOTH `devDependencies` and `overrides`:

```json
"vite": "npm:@voidzero-dev/vite-plus-core@<CORE_V>",
"vite-plus": "<CLI_V>",
"vitest": "npm:@voidzero-dev/vite-plus-test@<TEST_V>"
```

Use the exact version (no caret) to fully freeze, OR a caret (`^<V>`) if the team prefers patch flexibility — **prefer exact** here because these are `0.x` where caret still allows breaking `0.x.y` bumps. The `overrides` block must match the `devDependencies` specs exactly.

**Verify**: `grep -c "@latest\|\"latest\"" package.json` → returns 0.

### Step 3: Pin the specs in `sdks/typescript/package.json`

Apply the identical change in the SDK manifest (both `devDependencies` and `overrides`). Use the SDK's own resolved versions — check whether the SDK has its own lockfile or shares the root `bun.lock`. If the SDK resolves to the same versions, reuse `CORE_V`/`TEST_V`/`CLI_V`; if different, pin to the SDK's resolved versions.

**Verify**: `grep -c "@latest\|\"latest\"" sdks/typescript/package.json` → returns 0.

### Step 4: Regenerate and validate the lockfile

Run `bun install`. Because you pinned to the already-resolved versions, the lockfile should be unchanged or trivially updated.

**Verify**: `bun install --frozen-lockfile` → exit 0 (proves manifest and lockfile agree). Then `vp check` → exit 0 and `bun run test:unit` → tests pass (proves the pinned toolchain is functional).
