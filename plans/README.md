## How to use this directory

1. Pick a plan (respect the dependency graph below).
2. Re-read the plan's **Drift check** command first — the repo may have moved since `8ec01b0`. If cited code no longer matches, treat it as a STOP condition.
3. Follow the steps; run every verification command.
4. Update this file's **status table** row when done.

## Cross-cutting notes carried across plans

- **Drift stamp**: every plan is "Planned at commit `8ec01b0`, 2026-06-17" and opens with a `git diff --stat 8ec01b0..HEAD -- <files>` drift check. Run it first; mismatches are STOP conditions.
- **002 must be revisited if 017/018 ship**: the README "Current Limitations" section (no alerting, write-only SDKs) becomes inaccurate once alerting or read APIs land. Note this in those spikes' docs steps.
- **The `isNew`/reopen detection** for incidents is wanted by 012, 017, AND 019. Build it ONCE (in the upsert) and share it. 012 is the natural home; 017/019 consume it.
- **One status function**: 019 insists incident status be computed in exactly one place (`computeIncidentStatus`). Don't let 017's alerting re-derive status independently.
- **Shared query builder**: 018 extracts the logs-query builder so the session route and the new `/v1` read route share filters. 013's capped-count and 015's `parseLevelFilter`/`parseTimeRange` helpers should flow through the same shared code.

## Considered and rejected (or deferred)

These surfaced during the audit but were NOT turned into plans, with the reason. Recorded so they aren't re-litigated.

### Minor items, not separately planned

- **429 body-shape inconsistency**: `/v1/ingest` returns `{error, message}` while `/v1/logs` returns `{error}` for rate-limit responses. Cosmetic; plan 010 asserts both current shapes rather than normalizing. Fix opportunistically if touching both endpoints.
- **Timeline/timeseries final-bucket off-by-one** (BUG-03/BUG-04): the last time bucket can be computed slightly narrow. Minor visual effect on charts; not worth a dedicated migration-free change now. Revisit if charts show edge artifacts.
- **Health-endpoint version disclosure**: the health endpoint exposes a version string. Low value to hide for a self-hosted tool; not worth the change.

### Deferred pending investigation (not enough signal to plan safely)

- **Backfill orphan incidents** (admin-script territory): re-deriving incidents for pre-existing logs is an operational script (`incident-backfill.ts` exists), not a product change. Plan 020 reuses it for restore; standalone backfill deferred.
- **Login rate-limit IP-spoofing**: effectiveness depends on the reverse-proxy `X-Forwarded-For` configuration, which is deployment-specific. Investigate the proxy/trust config before changing app code — may be a docs item, not a code fix.
- **DEP-02 drizzle-kit config-load failure**: flagged but unverified against the CI platform. Verify whether it reproduces in CI before planning; may be a local-only environment artifact.

### Investigated and verified SAFE (no action)

- **IDOR / cross-tenant access**: `requireProjectOwnership` consistently returns 404 (not 403) to hide existence; verified across API routes. Plan 016 preserves this; plan 011 adds an explicit cross-tenant test.
- **XSS**: Svelte auto-escaping + no `{@html}` on user data in the audited paths.
- **SQL injection**: Drizzle parameterization throughout; raw `sql` uses are parameterized.
- **Constant-time comparison**: API-key verification uses hashing + appropriate comparison.
- **Mass assignment**: insert/update column sets are explicit, not spread from request bodies.
- **Dependency advisories**: the noisy advisories traced to an unreachable Prisma toolchain path (not in the runtime dependency graph). No action.

## Overall assessment

The codebase is **unusually mature and defensively coded**. The findings here are sharp edges — a live-event drop under SSE backpressure (006), a fallback auth secret (008), N+1 and over-fetch on the ingest hot path (012), an unbounded count (013) — not foundational gaps. The two genuinely high-risk plans (014 tsvector migration, 019 lifecycle schema change) are flagged as such and gated. The four direction spikes (017–020) target the product's real next frontier: turning a robust write-and-view log sink into a queryable, alertable, operable platform.
